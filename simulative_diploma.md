# Дипломный проект: Аналитика онлайн-маркетплейса

## Описание

Учебный проект курса Data Analytics. Реализована полная цепочка от загрузки данных через API до оперативного BI-дашборда:

- Ежедневная загрузка из API маркетплейса (cron на VPS)
- PostgreSQL 16 в Docker для хранения 8.7 млн строк за 4+ года
- Metabase в Docker для визуализации
- Дашборд "Marketplace Operations" с тремя разделами (KPI, активность клиентов, продукция)

Этапы проекта: разведка API → схема БД → скрипт загрузки → деплой на VPS с cron → BI-дашборд (текущий этап) → исследования (ABC×XYZ, RFM+когорты, в работе).

## Структура дашборда

**Вкладка "Основные показатели"** — оперативные метрики:
- KPI-плитки: Выручка за период, Выручка за день (Trend), Заказы, Уникальные клиенты, Средний чек, Медианный чек, Среднее количество позиций
- Графики динамики: Выручка по дням (с MA-7), Средний чек по дням

**Вкладка "Анализ активности клиентов"** — поведенческий анализ:
- DAU по дням
- Когортный анализ удержания (12 недель × 4 недели)
- Возвращающиеся vs Продолжающие клиенты по неделям
- Тепловая карта активности (день недели × час)

**Вкладка "Продукция"** — ассортимент:
- Топ-20 товаров по выручке

Глобальный фильтр периода: дефолт **Previous 90 days**, переопределяется в шапке дашборда.

## Замечания об особенностях данных

В процессе работы с дашбордом выявлены особенности синтетического датасета, использованного API:

1. **Структура клиентского пула.** ~99% уникальных клиентов имеют первую покупку в 2022 году. Новые клиенты в более поздние годы появляются редко (~5-10% в год). Это означает, что классическая метрика "новые vs повторные клиенты" вырождается. Метрика на дашборде переопределена как "Возвращающиеся (после паузы 4+ недель) vs Продолжающие".

2. **Распределение quantity.** Поле quantity распределено близко к равномерному на интервале 1-100, что нетипично для реального маркетплейса (где обычно 1-3). Это влечёт высокий средний чек (~825к ₽) и среднее количество позиций ~33.

3. **Стационарность временных рядов.** Выручка по дням, DAU и активность по часам не показывают трендов и сезонности — генератор API не моделирует поведенческие циклы. На реальных данных эти карточки показывали бы рост, недельную сезонность и пики активности.

Эти моменты не являются ошибками реализации, а отражены как аналитические находки.
